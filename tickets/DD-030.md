# DD-030

## Title

Repair the `browser` Node runtime bootstrap and prove it against the latest DD source.

## Method

TDD

## Problem

`dashboard browser.get https://example.com --script 'return document.title'` could fail on a real installed skill with:

- `Error [ERR_REQUIRE_ESM]: require() of ES Module .../uuid/dist-node/index.js`

The practical failure mode was that `playwright_server` loaded a broken `uuid` tree from `$HOME/node_modules`, so the skill could not even start a normal browser request.

## Cause

- the runtime dependency install path was not fully aligned with the latest DD package.json install model
- stale files from an older `uuid` install could survive in `$HOME/node_modules/uuid`
- the resulting mixed tree left `playwright_server` trying to `require('uuid')` from an incompatible install state
- a fresh DD-source verification path also needed a usable browser executable when `CHROMIUM_BIN` was unset

## Acceptance

- the skill repairs stale package installs instead of reusing a broken merged module tree
- runtime dependency install follows the latest DD staged `npx` package.json pattern
- direct host execution returns valid JSON for `browser.get https://example.com --script 'return document.title'`
- the same command works through the latest DD source checkout after loading DD and skill Perl dependencies
- Docker tests and Docker coverage both pass with `100%` statement and `100%` subroutine coverage for `lib/Browser/CLI.pm` and `lib/Browser/Runner.pm`

## Solution

- pin `uuid` to a CommonJS-safe major version in `package.json`
- stage package installation with `npx --yes npm install ...` in the DD cache
- clear the affected installed module directories before copying the fresh staged install into `$HOME/node_modules`
- add runtime fingerprint and dependency checks so the skill knows when to repair the install
- auto-detect a Chromium or Chrome executable from `PATH` when `CHROMIUM_BIN` is not set
- extend unit tests to cover the runtime repair and launch-option paths
